02 Why Plan Data

Every Screen Needs Data

Your wireframes show WHAT users see. But WHERE does that information come from? It comes from a database.

  • Home feed → needs posts/items data
  • Profile → needs user profile data

02 Fundamentals

What is a Database?

A database is an organized collection of data stored electronically. Think of it as a super-powered Excel spreadsheet.

In Supabase, your database uses PostgreSQL, the most powerful open-source database in the world.

Every app with dynamic content (content that changes) needs a database behind it.

04 Primary Key

Primary Key = Unique ID

Every table needs a primary key: a unique identifier for each record. No two records can have the same ID.

  • UUID — Random unique string (Supabase default). Best for most cases.
  • Auto-increment — 1, 2, 3, 4... Simple pero sequential.

In Supabase, the id column is automatically created as UUID. You don't need to manage it.

05 Relationships

Table Relationships

Tables don't live in isolation. They CONNECT to each other through relationships.

Example: An item is posted by a user. The items table needs to know WHICH user posted it.

This connection is made through a foreign key: a field in one table that references the primary key of another table.

05 1:Many

One-to-Many Relationship

ONE user can have MANY items.

How? The items table has a user_id field pointing to the user who created it.

users (1)

↓ has many

items (many)

Real examples:

  • 1 User → Many Posts
  • 1 User → Many Orders
  • 1 Category → Many Items
  • 1 Teacher → Many Students
  • 1 Classroom → Many Schedules

This is the MOST COMMON relationship in apps.

05 Foreign Key

How Foreign Keys Work

👤 users
id: abc-123PK
name: Maria
email: maria@...
📦 items
id: xyz-789PK
title: Black Umbrella
user_id: abc-123FK → users

The user_id in items matches the id in users. That's the link. Now we know Maria posted the umbrella.

06 M:M

Many-to-Many Relationship

Some relationships go BOTH ways:

  • Many students can enroll in many courses
  • Many users can bookmark many items
  • Many products can belong to many categories

Solution: create a junction table (also called a join table):

students ←→ enrollments ←→ courses
(student_id + course_id in the enrollments table)

06 Summary

Relationship Types

TypeExampleHow to Implement
One-to-One1 user → 1 profileuser_id in profile table (unique)
One-to-Many1 user → many postsuser_id in posts table
Many-to-Manystudents ↔ coursesJunction table with both IDs

For most student projects, One-to-Many is all you need. Keep it simple.

07 Auth

Supabase Handles Auth

Good news: you DON'T create a users table for login. Supabase has a built-in auth.users table that handles:

  • ✅ Email/password registration
  • ✅ Password hashing (security)
  • ✅ Session management (stay logged in)
  • ✅ Unique user IDs (uuid)

BUT: You'll create a separate profiles table for extra user data (name, avatar, role, bio).

07 Profiles Pattern

The Profiles Table

👤 profiles
iduuid (= auth.users.id)
full_nametext
avatar_urltext
roletext (student/admin)
student_idtext
created_attimestamp

Key pattern:

  • The id matches the auth user's ID
  • Created when user signs up
  • Stores display name, avatar, role
  • All other tables reference this ID

This is the STANDARD pattern for Supabase apps. Almost every project uses it.

09 Status Pattern

Status Fields

Most apps need a STATUS field to track where something is in its lifecycle:

Lost & Found

lost → found → claimed

Order System

pending → processing → completed

Task Manager

todo → in_progress → done

Store as text with predefined values. In FlutterFlow, display with colored badges.

09 Queries

Think About Queries

Before finalizing your schema, ask: "What questions will my app ask the database?"

  • "Show me all lost items posted this week" → filter by status + created_at
  • "Show me items posted by this user" → filter by user_id
  • "Show items in the Electronics category" → filter by category_id
  • "How many items are currently unclaimed?" → count where status ≠ claimed
  • "Show the newest items first" → order by created_at DESC

If you can't answer a question with your current fields, you need to add one!

10 Process

Schema Planning Steps

Step 1: List all "things" in your app (nouns) → tables
Step 2: For each table, list properties → fields
Step 3: Choose data types for each field
Step 4: Add id, created_at, user_id to every table
Step 5: Identify relationships (who owns what?)
Step 6: Add foreign key fields
Step 7: Check: can my screens query what they need?

10 Avoid

Database Mistakes

  • No user_id — Can't tell who created what. Always link records to users.
  • Storing lists in one field — Don't put "cat1,cat2,cat3" in one text field. Use a separate table.
  • No timestamps — You'll always need to know WHEN something was created.
  • Too many tables — Start simple. 2-4 tables is fine for MVP. Add more later.
  • Wrong data types — Don't store numbers as text. Don't store dates as text.
  • No status field — Almost every data entity has a lifecycle. Track it.
10 MVP Schema

Minimum Schema for MVP

For most student projects, you need exactly:

👤 profiles
id, name, role
📦 main_entity
id, fields, user_id, status, created_at
🏷️ categories
id, name

That's it. 2-3 tables. Don't over-engineer. You can always add more tables later.
