02 Definition

What is a Wireframe?

A wireframe is a simple sketch of a screen's layout. No colors, no images, no details. Just boxes, lines, and labels showing WHERE things go.

Think of it as the blueprint of a house. Not the final paint and furniture, just the walls and room positions.

02 Purpose

Why Wireframe First?

Benefits

  • Cheap to change (just erase!)
  • Focus on LAYOUT not looks
  • Catch problems early
  • Plan before you build
  • Takes minutes, not hours

Without wireframes:

  • Open FlutterFlow clueless
  • Stare at blank canvas
  • Redesign 5 times
  • Forget elements
  • Waste hours experimenting
02 Traceability

From Requirement to Rectangle

Start with the requirement

“The user can report a lost item.”

This tells us the capability, but not yet the screen structure.

Translate it into layout

Report form → title → item fields → photo area → location → submit button.

The wireframe makes every required input and action visible before development.

Requirement → screen purpose → content hierarchy → primary action → next screen

Today we do LOW-FIDELITY. Quick, messy, and useful. Perfection is the enemy.

02 Decision Guide

What Detail Belongs When?

Low fidelity

Screen purpose, rough regions, labels, navigation, and major states.

Question: Does the flow make sense?

Mid fidelity

Consistent spacing, realistic content, component sizing, and clearer interactions.

Question: Is the layout usable?

High fidelity

Brand colors, typography, imagery, component states, and near-final behavior.

Question: Is this ready to build?

Do not spend high-fidelity effort on a flow that has not passed low-fidelity review.

06 Rules

Wireframe Do's & Don'ts

✅ DO

  • Use simple shapes (boxes, lines)
  • Label everything
  • Show real content structure
  • Draw ALL screens
  • Keep proportions realistic
  • Mark required fields

❌ DON'T

  • Add colors or styling
  • Use real images
  • Worry about pixel perfection
  • Spend 30 min on one screen
  • Forget navigation elements
  • Skip error/empty states
06 Process

How to Start Wireframing

Step 1: Get your screen list from your earlier work
Step 2: Draw a browser rectangle (wide landscape)
Step 3: Add the header (app bar) at top
Step 4: Add navigation (bottom tab or drawer)
Step 5: Fill the body with elements (inputs, cards, buttons)
Step 6: Label everything clearly
Step 7: Move to next screen. Repeat.

06 Layout Tips

Layout Tips

  • Content should not touch edges — always leave margin (padding)
  • Group related elements — label next to input, not floating
  • Primary action at bottom — easy to reach with mouse, keyboard, or touch
  • One primary action per screen — what's THE thing to do here?
  • Scroll direction: vertical only — horizontal scroll confuses users
  • Above the fold matters — most important content visible without scrolling
07 Often Forgotten

Empty States

What does your screen look like when there's NO DATA?

  • No items posted yet
  • No search results
  • No notifications
  • First time user (nothing saved)

Always design the empty state! It's the first thing new users see.

07 Edge Cases

Error & Loading States

⏳

Loading

Spinner or skeleton when fetching data

⚠️

Error

Clear message + retry button

✅

Success

Confirmation message or redirect

Pro move: sketch these states too. It shows you've thought about the FULL experience.

07 Complete Experience

Design the States, Not Only the Happy Path

Before data

Loading indicators or skeletons explain that the app is working.

No data

An empty state explains what is missing and suggests the next useful action.

After failure

An error state says what happened and offers a retry or recovery path.

One “Home” screen may need separate wireframes for loading, empty, content, and error conditions.

08 Annotations

Annotate Your Wireframes

  • Number each screen — "Screen 1: Login", "Screen 2: Home"
  • Label all elements — What is this box? A button? An image?
  • Note interactions — "Click → opens Page 3"
  • Mark scrollable areas — Arrow down = this area scrolls
  • Note dynamic content — "This list loads from database"
  • Indicate required vs optional — Star (*) for required fields

Your wireframe should be understandable even if someone else reads it.

09 Navigation Flow

Connecting Screens

After wireframing individual screens, show HOW they connect:

  • Draw arrows between screens
  • Label what triggers navigation ("Click Login → Home")
  • Show the back navigation (can user go back?)
  • Indicate modal vs full-screen navigation

This becomes your navigation map (we'll formalize it next session).

09 Checklist

Wireframe Checklist

Per Screen:

  • ☐ Screen title/number
  • ☐ Header/app bar
  • ☐ All content elements
  • ☐ Navigation (where to go)
  • ☐ Primary action clear
  • ☐ All elements labeled

Overall:

  • ☐ All screens from screen list drawn
  • ☐ Consistent navigation across screens
  • ☐ Empty state for list screens
  • ☐ Error states considered
  • ☐ Arrows showing screen connections
  • ☐ User can complete core task visually
10 Scope

How Many Wireframes?

A small app usually needs 3-5 core screens to explain its main journey:

  • 1. Login/Register — everyone needs this
  • 2. Home/Main screen — the landing page
  • 3. Create/Form screen — where users input data
  • 4. Detail screen — view one item in detail
  • 5. Profile/Settings — user account

For a small MVP, these five screens usually explain the core user journey.

10 Recap

What We Covered

  • Wireframes = simple layout sketches (no colors, no details)
  • Low-fidelity is fast and cheap to change
  • Standard elements: boxes, labels, arrows, X-for-images
  • Every screen has: header, body, navigation
  • Don't forget empty states and error states
  • A small MVP usually needs 3-5 core wireframes
  • Arrows show how screens connect
