02 Why Patterns

Why Learn Screen Patterns?

You don't invent a new steering wheel every time you build a car. Screen patterns are proven solutions that users already understand.

  • Users already know how to use them
  • Reduces confusion and learning curve
  • Faster to build (patterns map directly to FlutterFlow widgets)
  • Professional-quality results
02 Pattern 1

Login / Auth Screen

Standard Elements

  • App logo/brand at top
  • Welcome/greeting text
  • Email + password fields
  • Login button (primary CTA)
  • Forgot password link
  • Sign up link
  • Optional: social login (Google)

Variations

  • Split: Login and Register as separate pages
  • Tabbed: Login | Register links on one page
  • Onboarding first: 2-3 swipeable intro screens before login
  • Guest mode: "Skip" option to browse without account
02 Best Practices

Auth Screen Rules

  • Minimal fields — Only ask what's needed (email + password). No phone/address at login.
  • Show/hide password toggle — Let users see what they typed
  • Clear error messages — "Invalid email format" not "Error 401"
  • Remember me option — Users hate re-logging every time
  • Loading state on button — Show spinner while authenticating
  • Keyboard-aware — Inputs shouldn't hide behind keyboard
03 Pattern 2

Home / Dashboard

Purpose

The landing page after login. Answers two questions:

  • "What's new?" — Recent content/activity
  • "What can I do?" — Quick access to actions

Common Layouts

  • Feed: Cards/posts (social apps)
  • Dashboard: Stats + quick actions (utility)
  • Grid: Categories/products (e-commerce)
  • Map: Location-first (ride/delivery)
03 Anatomy

Home Screen Anatomy

Header: App name, notification bell, avatar/menu
Search/Filter: Quick way to find content (optional)
Hero Section: Featured content, stats card, or banner
Quick Actions: 2-4 shortcut buttons for common tasks
Content Feed: Scrollable list/grid of items
Primary Navigation: Persistent top bar or sidebar for main sections

Not every home screen has ALL of these. Pick what fits YOUR app.

04 Pattern 3

Form / Create Screen

Elements

  • Page title ("Create Post")
  • Back/Cancel button
  • Labeled input fields
  • Dropdowns/selectors
  • Date/time pickers
  • Image upload area
  • Submit button (sticky bottom)

Form UX Rules

  • Labels above fields (not placeholder-only)
  • Mark required fields (*)
  • Show inline validation errors
  • Disable submit until form is valid
  • Confirm before discarding unsaved data
  • Success feedback after submit

05 Pattern 4

List / Feed Screen

List Variations

  • Simple list: Text only rows (settings)
  • Image + text: Thumbnail + title + subtitle
  • Card list: Full cards with multiple data
  • Grid: 2-3 columns of items

Must-Have Features

  • Pull-to-refresh
  • Loading skeleton/spinner
  • Empty state message
  • Tap item → Detail screen
  • Optional: search, filter, sort

06 Pattern 5

Detail / View Screen

Structure

  • Back button (return to list)
  • Hero image (full-width)
  • Title + category badges
  • Full description
  • Metadata (date, author, location)
  • Primary action button
  • Optional: edit/delete menu

Design Tips

  • Image FIRST, info below
  • Clear hierarchy: title → description → meta
  • Sticky action button at bottom
  • Allow scrolling for long content
  • Show loading state while fetching
07Web Navigation

Navigation Patterns for the Web Project

▰

Top Bar

Best for a small set of main destinations and one visible primary action.

▥

Sidebar

Keeps several sections visible on laptop screens and can collapse at narrow widths.

→

Page Routing

List → detail → edit preserves a clear Back path and browser history.

The Lost & Found example uses a persistent sidebar/top bar plus routed detail pages.

07Primary Navigation

Persistent Web Navigation

Rules for the course project:

  • Keep 3–6 main destinations visible. Use Home, Browse, My Items, and Profile.
  • Use text labels. Icons may support labels but should not replace them.
  • Show the current page. Highlight the active link and keep navigation placement consistent.
  • Separate the primary action. “Report Item” belongs in a visible header button.
  • Adapt at narrow widths. Collapse the sidebar or wrap the top bar without changing the information architecture.
07 Stack Nav

Stack Navigation

Pages layer on top of each other like a stack of cards:

Home (base)
↓ tap item
Detail Page (pushed on top)
↓ tap edit
Edit Form (pushed on top)
← back = pop one level
← back = pop to Home

FlutterFlow handles this automatically when you set "Navigate To" actions.

08 Drawer

Drawer Menu

When to Use

  • 6+ navigation sections
  • Less-frequent navigation items
  • Admin/settings pages
  • Secondary features

Drawer Contents

  • User avatar + name (top)
  • Main navigation links
  • Dividers between groups
  • Settings / Help / Logout (bottom)

Pro tip: You can combine Tab Bar (main sections) + Drawer (secondary/settings). Many apps do this.

09 Nav Map

Building a Navigation Map

A navigation map shows your entire app's page structure and how users move between them.

It's like a sitemap but for responsive web apps. Shows:

  • All screens organized by section
  • Navigation type (top bar, sidebar, and routed pages)
  • Which screens link to which
  • User role differences (if any)

09 Patterns

Navigation Patterns

Auth Guard Pattern

App checks: logged in?

Yes → Home

No → Login page

FlutterFlow handles this with "Authenticated" routes

Role-Based Navigation

After login, check user role:

Student → Student Home

Admin → Admin Dashboard

Different primary navigations or different pages per role

Modal/Popup Pattern

Quick actions that overlay current screen:

Confirm delete, filter options, quick add

Deep Link Pattern

User taps notification → jumps directly to a specific detail page

10 Critical Rule

Never Trap Users

Users must ALWAYS be able to go back or escape:

  • ✅ Every non-home screen has a back button
  • ✅ Android back gesture/button works
  • ✅ Primary web navigation is visible on every main page
  • ✅ Forms have a cancel/close option
  • ❌ NEVER create a dead-end page with no way out

11 Avoid

Navigation Mistakes

  • Too many links — More than 5 links overwhelms. Use drawer for extras.
  • Inconsistent nav — Primary navigation appears/disappears unpredictably
  • Deep nesting — User is 5 levels deep, lost. Max 3 levels recommended.
  • No back button — User can't return to previous page
  • Confusing icons — Icon-only links with no labels
  • Duplicate paths — Same screen reachable from 3 different places (confusing)
11 Preview

Navigation in FlutterFlow

When we start building, FlutterFlow gives you:

  • NavBar widget — Persistent web navigation with icon + label + routes
  • Drawer widget — Side menu with user info + links
  • "Navigate To" action — Stack push (with transition options)
  • "Navigate Back" action — Stack pop
  • "Replace" action — Swap without stack (login → home)
  • Route guards — Require authentication for certain pages

Your navigation map today → directly translates to FlutterFlow setup later.
