02 Development Approaches

Native vs Cross-Platform

Native

Built for ONE platform only

  • iOS → Swift/Objective-C
  • Android → Kotlin/Java
  • Best performance
  • Higher cost (2 codebases)

Cross-Platform

ONE codebase, multiple platforms

  • Flutter (Dart)
  • React Native (JavaScript)
  • Good performance
  • Faster development 🏆
02 In Plain Language

The app type tells you where it runs

Platform

A platform is the environment where people use the app: a phone, a browser, or a desktop computer.

The same service can have several versions. Gmail, for example, has web and mobile apps.

Why cross-platform?

A cross-platform tool lets one project target more than one platform.

For this course, that means you can focus on solving the problem instead of maintaining separate Android, iOS, and web projects.

Key idea: cross-platform saves development time, but you still test the app on every target device.

03 Under The Hood

How Apps Actually Work

USER → Frontend (UI) → API → Backend (Server) → Database

Every app follows this pattern. The user interacts with the frontend. The frontend talks to the backend. The backend stores/retrieves data.

03 Worked Example

One tap can trigger a complete data journey

  1. You tap "Log in" on the frontend.
  2. The frontend sends your email and password through an API request.
  3. The backend checks the stored account information in the database.
  4. The backend returns a success or error response.
  5. The frontend shows the home screen or an error message.

The user sees one action, but several parts cooperate behind the scenes.

03 Architecture

Frontend vs Backend

Frontend

What users SEE and TOUCH

  • Buttons, screens, forms
  • Colors, fonts, layout
  • Animations, navigation
  • Our tool: FlutterFlow

Backend

What runs BEHIND the scenes

  • Database (stores data)
  • Authentication (login)
  • Business logic, APIs
  • Our tool: Supabase
03 Key Concept

What is an API?

API = Application Programming Interface

Think of it as a waiter in a restaurant:

  • You (frontend) tell the waiter your order
  • The waiter (API) brings it to the kitchen (backend)
  • The kitchen prepares the food (processes data)
  • The waiter brings back your food (response)
03 API Vocabulary

A request asks; a response answers

Request

The app sends an instruction or asks for data.

Example: "Give me all lost-item posts added today."

Response

The backend returns data or explains what went wrong.

Example: a list of posts, or an error saying the connection failed.

APIs use agreed rules so two systems can exchange information without knowing each other's internal code.

04 Who Does What?

FlutterFlow builds the experience; Supabase manages the data

FlutterFlow

  • Screens and layouts
  • Buttons, forms, and navigation
  • Actions that send or request data
  • What the user sees after a response

Supabase

  • User accounts and login
  • Database tables and records
  • Files and uploaded images
  • Rules controlling who can access data

Connection: FlutterFlow calls Supabase APIs, then turns the returned data into screens the user can understand.
